### Step 4: 아키텍처 다이어그램

**Agent에게 요청**:

```
마이크로서비스 아키텍처 다이어그램 만들어줘.

구성:
- 상단: Client (브라우저, 모바일 앱)
- 중간: API Gateway
- 하단: 서비스들
  - Auth Service (인증)
  - User Service (사용자 관리)
  - Order Service (주문 처리)
  - Payment Service (결제)
- 각 서비스마다 Database 연결
- 중앙: Message Queue (RabbitMQ)
- 우측: Cache (Redis)

화살표로 연결 관계 표시:
- Client → API Gateway
- API Gateway → 각 Service
- Service → Database
- Service ↔ Message Queue

스타일:
- 박스: 둥근 모서리
- 색상: 각 레이어별로 다른 색상
- 텍스트: 명확하게 읽을 수 있게

파일명: assets/diagrams/architecture.png
```

**확인 사항**:
- [ ] 모든 컴포넌트가 표시되었는가?
- [ ] 연결 관계가 명확한가?
- [ ] 이해하기 쉬운가?

### Step 5: 사용자 플로우차트

**Agent에게 요청**:

```
회원가입 플로우차트 만들어줘.

단계:
1. 시작
2. 이메일 입력
3. 이메일 유효성 검사
   - 유효하지 않음 → 에러 메시지 → 2번으로
   - 유효함 → 4번으로
4. 비밀번호 입력
5. 비밀번호 확인
   - 일치하지 않음 → 에러 메시지 → 4번으로
   - 일치함 → 6번으로
6. 이메일 인증 발송
7. 인증 코드 입력
8. 인증 확인
   - 실패 → 에러 메시지 → 7번으로
   - 성공 → 9번으로
9. 회원가입 완료
10. 끝

스타일:
- 시작/끝: 타원
- 프로세스: 사각형
- 판단: 다이아몬드
- 화살표로 연결
- 색상: 성공 경로는 초록색, 에러 경로는 빨간색

파일명: assets/diagrams/signup-flow.png
```

**확인 사항**:
- [ ] 모든 단계가 포함되었는가?
- [ ] 분기 처리가 명확한가?
- [ ] 흐름을 따라가기 쉬운가?

### Step 6: ER 다이어그램

**Agent에게 요청**:

```
블로그 시스템 ER 다이어그램 만들어줘.

테이블:
1. Users
   - id (PK)
   - email
   - name
   - created_at

2. Posts
   - id (PK)
   - title
   - content
   - user_id (FK → Users.id)
   - created_at
   - updated_at

3. Comments
   - id (PK)
   - content
   - post_id (FK → Posts.id)
   - user_id (FK → Users.id)
   - created_at

4. Tags
   - id (PK)
   - name

5. PostTags (중간 테이블)
   - post_id (FK → Posts.id)
   - tag_id (FK → Tags.id)

관계:
- Users 1:N Posts
- Users 1:N Comments
- Posts 1:N Comments
- Posts N:M Tags (through PostTags)

스타일:
- 테이블: 사각형 (3개 영역: 이름, 필드, 키)
- 관계선: 1:N, N:M 표시
- PK: 노란색 배경
- FK: 파란색 배경

파일명: assets/diagrams/blog-er.png
```

**확인 사항**:
- [ ] 모든 테이블이 표시되었는가?
- [ ] 관계가 정확한가?
- [ ] 키 구분이 명확한가?

## 💡 실전 활용

```
You: 프로젝트 전체 화면 구성 목업 만들어줘:
     - 랜딩 페이지
     - 로그인/회원가입
     - 메인 대시보드
     - 설정 페이지

Agent: [4개 화면 목업 생성]
       
       → 기획 회의에서 사용
       → 클라이언트 프레젠테이션
```

### 2. 문서화

```
You: README.md에 사용할 아키텍처 다이어그램 만들어줘

Agent: [다이어그램 생성]
       README.md에 이미지 링크 추가했습니다
       
       → 프로젝트 문서 완성
```

### 3. 디자인 시스템

```
You: 디자인 시스템 컴포넌트 목업 만들어줘:
     - 버튼 (5가지 변형)
     - 입력 필드 (3가지 상태)
     - 카드 (2가지 스타일)
     - 모달 (2가지 크기)

Agent: [컴포넌트 목업 생성]
       
       → 개발 가이드로 사용
```

## 🎓 정리

### Image Generation의 가치

1. **빠른 시각화**: 몇 초 만에 목업 생성
2. **의사소통 개선**: 시각 자료로 명확한 전달
3. **문서화 강화**: 다이어그램으로 이해도 향상
4. **비용 절감**: 디자이너 없이도 프로토타입 제작

### 실전 팁

- 목업은 완벽할 필요 없음 (프로토타입 목적)
- 다이어그램은 명확성이 최우선
- 여러 버전을 만들어 비교
- 실제 프로젝트에 바로 적용
